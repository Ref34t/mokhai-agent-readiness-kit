# AgDR-0069 — Rendered-HTML → Markdown fallback (loopback source of last resort)

> In the context of **theme-template-rendered content (ACF Pro flexible-content, Sage/Acorn, and any builder that renders in templates rather than through `the_content`) yielding empty Markdown twins that neither `the_content` nor the ACF adapter (AgDR-0068) can recover on real agency sites (#297)**, facing **content that exists only in raw postmeta and the front-end render — unreachable via the ACF API because theme-registered sub-field definitions don't resolve in a plugin context** — we decided **to add a last-resort source adapter that loopback-fetches the post's own rendered front-end HTML, isolates the main-content region with a self-contained DOMDocument extractor (no new runtime dependency), and feeds it through the existing `mokhai_markdown_source_html` seam and Walker**, to achieve **non-empty, theme- and builder-agnostic Markdown twins on exactly the site class the plugin targets**, accepting **a loopback HTTP round-trip on cache-miss renders of empty-bodied pages, heuristic main-content extraction, and reliance on post-save invalidation (a pure template edit won't invalidate the cache until a walker-version bump or manual regenerate).**

## Context

- **#297 disproved the AgDR-0068 premise.** Live verification on a real ACF Pro site (Bedrock, niu-theme / Acorn-Sage, 21 pages) showed the empty-twin *guard* works (18/21 → `empty_content`, correctly excluded from `/llms.txt`) but the ACF *adapter* recovered content for **zero** pages. `get_field_objects()` returns flexible-content rows with only `acf_fc_layout` and no sub-field values, because the theme's sub-field definitions (keys `field_ic_*`, `field_mc_*`) aren't matched in a plugin context. The content is in raw postmeta and renders in the template, but never surfaces through the ACF API **or** `the_content`.
- The one place that content reliably exists in a resolved, theme-agnostic form is the **front-end HTML render itself** — the thing an AI agent fetching the page would actually see. AgDR-0068 named this "Option C" and deferred it as the "builder-agnostic successor."
- **A reviewed loopback pattern already exists in this module.** `Cache_Posture::refresh_probe_snapshot()` (AgDR-0067) already does `wp_remote_get(home_url('/'), …)` with a bounded timeout, redirection cap, size limit, and a self-identifying user-agent, and handles the `CaseInsensitiveDictionary` header shape. Option C reuses that plumbing shape rather than inventing loopback from scratch, which materially lowers its risk.
- **The Walker already cleans.** `Walker` (version 5, #253/AgDR-0063) drops `<script>`/`<style>` subtrees and strips base64 page-builder blobs before converting to Markdown. So the new extractor only has to *isolate the main-content region* — it does not need to re-implement sanitisation.
- **Zero runtime dependencies is a standing posture.** `composer.json require` is `php >=7.4` only. Pulling in a Readability library (`fivefilters/readability.php`) would add a bundled vendor tree and expand the wp.org review surface. A self-contained DOMDocument heuristic keeps the posture.

## Options Considered

| Option | Pros | Cons |
|--------|------|------|
| **C — rendered-HTML loopback + self-contained DOMDocument extractor** | Theme/builder-agnostic — captures ACF, Sage, and any template-rendered content in one mechanism. Reuses the AgDR-0067 loopback plumbing and the existing Walker sanitiser. No new runtime dependency. Last-resort, so zero cost on pages that already render content. | Loopback HTTP round-trip on cache-miss renders of empty pages. Main-content extraction is heuristic. Template-only edits don't invalidate the cache. |
| C-lib — same, but with a bundled Readability library | Battle-tested extraction quality. | New bundled vendor dep; larger wp.org review surface; breaks the zero-runtime-dep posture; still needs the loopback. |
| B — raw-postmeta pattern extraction (`{field}_{i}_{subfield}`) | Cheap, no HTTP. | Reintroduces exactly what AgDR-0068 rejected: no type metadata → dumps IDs / colours / select-keys as body noise. Fragile against ACF meta-key conventions. ACF-only (misses non-ACF template builders). |
| A — guard-only (status quo of #295) | Honest, already shipped. | Delivers **zero** content on the target site class — abandons the core promise for the exact sites Mokhai is pitched at. |

## Decision

Chosen: **Option C** (self-contained extractor, **not** the bundled-library variant), because it is the single mechanism that recovers the whole template-rendered class — not just ACF — it reuses two already-reviewed subsystems (AgDR-0067 loopback, AgDR-0063 Walker sanitiser), and it preserves the zero-runtime-dependency posture. Option B is a fragile, noise-prone dead end; Option A abandons the value proposition on the target sites.

The ACF adapter (AgDR-0068) is **kept** as a cheaper first-choice source: on sites where `get_field_objects()` *does* resolve, it avoids the HTTP round-trip. Loopback sits below it as the theme-agnostic net. #297's "the adapter may be removable" is resolved as **keep-as-fast-path**.

### Shape

- **`Markdown_Views/Rendered_Html_Source`** — a new bundled adapter mirroring `Woocommerce_Source` / `Acf_Source`, registered unconditionally from `Main::register_hooks()`, hooking `mokhai_markdown_source_html` at a **late priority (20)** so it runs *after* the ACF adapter (10). It no-ops unless the incoming `$html` is still empty after all earlier sources ran — so it is genuinely last-resort and never duplicates content.
  - **Fetch — loopback GET of `get_permalink($post)`** using the `Cache_Posture` request shape: bounded `timeout` (5s), `redirection` cap (2), a `limit_response_size` ceiling (2 MB — enough for a page, bounds memory), a self-identifying `user-agent` (`mokhai-render-fetch/1.0`), and `reject_unsafe_urls => true` for SSRF defence-in-depth (S3).
  - **HTTP-200 gate (B2, blocking)** — `wp_remote_get` does **not** return a `WP_Error` for 401/403/500; it returns a normal response object carrying the error page's body. A staging basic-auth wall (401 login page) or the #256 fixture's own WeglotAddon fatal (500 WSOD) would otherwise be *extracted and cached as the twin* — a confidently-wrong body, strictly worse than the empty guard. The adapter therefore asserts `wp_remote_retrieve_response_code() === 200` **and** a non-empty body before extraction; any other status → return `$html` unchanged → empty guard fires → 404.
  - **Same-host assertion** — only fetch when the permalink's host equals `home_url()`'s host. Never fetch cross-host. This is a fetch of the site's *own* public page as an anonymous visitor, so it sees exactly what an agent would see (correct semantic; no privilege escalation). Note: `redirection => 2` follows redirects internally, so the same-host check constrains the *initial* URL, not the redirect target (the site's own redirect config — low risk; `reject_unsafe_urls` is the backstop) (S3).
  - **Cross-process recursion guard (B1, blocking) — a DB-backed lock, not user-agent.** The loopback is a *separate* HTTP request served by a *separate* PHP-FPM worker, so any in-process/static flag is invisible to it, and a WAF/security plugin can strip a non-browser user-agent before it reaches PHP (guard fails open). The load-bearing guard is therefore a short-TTL post-scoped lock: `set_transient('mokhai_render_lock_' . $post_id, 1, 30)` is set **before** the fetch and checked at the top of the adapter — cross-process because the DB is shared, and it doubles as cache-stampede dedup when two agents hit the same empty `.md` concurrently. A custom **in-request marker** (query arg / header the loopback worker reads to no-op) is the secondary guard; user-agent is a tertiary hint only. The lock is released with `delete_transient()` on completion of the render (both the extract-and-cache success path and the failure path), not left to expire — so a legitimate re-render within the TTL window is not silently starved; the 30s TTL is only the crash-safety net. **Ordering constraint:** this guard MUST land in the same release as — or before — #296. #296 makes `Alternate_Advertiser` call `is_empty_for_post()` at `wp_head`/`send_headers` time; without the lock, a loopback GET → HTML render → `wp_head` → `is_empty_for_post()` → cache-miss → loopback@20 → GET again becomes a blocking loopback storm that exhausts PHP workers (self-inflicted DoS), not merely a nested fetch. (Today the chain is *not* live: `Alternate_Advertiser` does not call `Service` on an HTML render — verified — so the recursion is latent until #296 activates it.)
  - **Main-content extraction — self-contained DOMDocument, no new dependency** (`Rendered_Html_Source::extract_main_html()`, pure over an HTML string so it is unit-testable without a live site). The parser MUST reuse the Walker's existing DOMDocument hardening (UTF-8 encoding hint, `libxml_use_internal_errors()` / `LIBXML_NOERROR`) rather than re-deriving it, so malformed HTML5 doesn't emit warnings or mis-parse (N1):
    1. Strip chrome subtrees: `<nav> <header> <footer> <aside> <form> <script> <style> <noscript>` and elements with `role` in `navigation|banner|contentinfo|search|complementary`.
    2. Semantic-first region pick: first `<main>`, then `<article>`, then `[role=main]`, then a small ordered list of common content containers (`.entry-content`, `#content`, `#primary`, `.site-content`).
    3. Density fallback: if no semantic region matches, score block subtrees by text-length-to-markup ratio and pick the densest.
    4. Return the region's inner HTML — the existing **Walker** then does script/style/blob stripping and the HTML→Markdown conversion. The extractor isolates; the Walker cleans and converts.
    - **Residual extraction risk (N2):** the density fallback (step 3) is the most likely to misfire — a text-heavy sidebar, or a cookie/consent banner not caught by the `role`/tag chrome list of step 1, can win. Chrome-strip running first mitigates it; consent-banner containers are the residual risk the #256 fixture must exercise before the heuristic-vs-library escalation is judged.
- **Chain & guard interplay** — the empty-twin guard (AgDR-0068) stays as the final backstop: pages that even the loopback can't source (fetch failure, genuinely empty page) still 404 + drop from `/llms.txt`. Loopback *reduces* the empty set; the guard covers the remainder. (This is the direct upstream of #296 — see below.)
- **Caching / invalidation** — the loopback fetch happens **only on a cache-miss render** inside `Service::get_markdown_for_post()`, never on the `content_hash` path. So there is **no per-request HTTP** (contrast AgDR-0068 F3, which reintroduced an ACF read on every request — this design does not repeat that; the empty-string result is itself a valid cache entry, so a genuinely-empty page fires loopback once per content-hash, not per request). Invalidation rides the existing `save_post` / `wp_after_insert_post` hooks and `post_modified_gmt`. **Accepted limitation:** a pure theme/template edit with no post save won't invalidate the cached render until a `WALKER_VERSION` bump or a manual regenerate. **The daily mirror-sync cron is NOT a backstop for this** (S1): `Static_Mirror::is_file_fresh()` compares `filemtime` against `post_modified_gmt`, and a template edit doesn't bump `post_modified_gmt`, so the cron treats the file as fresh and skips it. Accepted for a text twin; documented so no reader assumes an automatic backstop that doesn't exist.
- **Static mirror coverage** — `Static_Mirror` (AgDR-0067) renders through the same `Service` path, so it inherits the loopback source automatically (the #293/#296 "must cover the static mirror too" lesson). The publish-time mirror write of an empty-bodied page will incur one loopback fetch — acceptable, it runs in an admin/background context and only for empty pages.
- **Cold-start / bulk cost bound (S2)** — `Static_Mirror::run_daily_sync()` processes up to `SYNC_BATCH_CAP` (50) posts per tick via `get_markdown_for_post()`. On first activation / bulk import, up to 50 empty-bodied cache-misses could each incur a 5s-timeout loopback (~250s of blocking fetches in one tick). Steady state is fine (`is_file_fresh` skips written files). The adapter therefore caps loopbacks-per-tick and/or uses a shorter timeout in the cron/background context; the remainder is picked up on subsequent ticks.
- **Config** — filter-gated `mokhai_markdown_loopback_enabled` (default `true`); honours a Context Profile `rendered_html_fallback` toggle if set. A hard-off environment (loopback blocked by host firewall) degrades gracefully: the fetch `WP_Error`s → the adapter returns `$html` unchanged → the empty guard fires. No fatal, no hang beyond the 5s timeout.

## Consequences

- Template-rendered pages (ACF-in-templates, Sage, and non-ACF template builders) get non-empty, agent-faithful Markdown twins — the recovery mechanism is now what the page actually renders, not what an API happens to expose.
- One loopback HTTP round-trip per cache-miss render of an empty-bodied page. Bounded (5s timeout, 2 MB cap), off the hot path (cache hits pay nothing), and only for pages that would otherwise be empty.
- Main-content extraction is heuristic — a badly-structured theme may over- or under-capture. Non-empty, slightly-imperfect content beats a 0-byte body; a bundled Readability library remains the escalation path if the heuristic proves inadequate on the #256 fixture.
- A pure template/theme edit won't invalidate a cached twin (see Caching, above). Acceptable for a text twin; documented.
- The ACF adapter (AgDR-0068) stays as the no-HTTP fast path; the two adapters + the base source now form an ordered chain (`the_content` → ACF@10 → loopback@20 → empty-guard).
- **Live E2E verification is gated on a working fixture, not the broken staging site (S4).** The niu-theme staging site is blocked by a pre-existing WeglotAddon fatal (`getOriginalLanguage()`, a site bug, not Mokhai's) that 500s every request — and per B2 the adapter would now correctly refuse to extract that 500. Unit tests over fixture HTML cover the extractor and the chrome/density logic. Loopback E2E runs against the **#256 messy-site CI fixture** (Woo + page-builder + ACF flexible-content) — the concrete, self-owned target that doesn't depend on the one external site that 500s. The value-prop claim ("non-empty twins on the target site class") ships as *proven* only once #256 E2E passes; until then it is *designed-for*, not *verified*.

## Relationship to #296

#296 (the `Alternate_Advertiser` advertising a `.md rel=alternate` that 404s for empty twins) is a **fast-follow, still needed after this lands**. Loopback shrinks the empty-twin set but does not eliminate it (fetch failures, genuinely-empty pages). The advertiser must not advertise a twin the guard will 404 — that fix is orthogonal to content recovery and is tracked separately.
